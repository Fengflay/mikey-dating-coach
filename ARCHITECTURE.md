# Mikey情感导师 - 系统架构总览

## 项目结构

```
mikey-dating-coach/
├── ARCHITECTURE.md              # 本文件
├── prompts/
│   ├── system_prompt_main.md    # 主系统提示词 (角色定义+分析框架+回复规则)
│   ├── dynamic_context_template.md  # 动态上下文注入模板 (Handlebars风格)
│   └── analysis_prompt_template.md  # 两阶段分析+生成提示词模板
├── knowledge-base/
│   └── rag_architecture.md      # 知识库RAG架构 (分类体系+分块策略+检索策略)
├── schemas/
│   ├── session_management.md    # 会话管理设计 (数据模型+上下文窗口+历史压缩)
│   └── example_output.md        # 完整分析输出示例
└── scripts/
    ├── input_processing.py      # 输入处理模块 (OCR+文本解析+元数据提取)
    └── orchestrator.py          # 编排引擎 (两阶段pipeline+RAG集成)
```

## 数据流

```
用户输入 (文本粘贴 / 截图)
       │
       ▼
┌──────────────────┐
│  输入解析模块     │  正则预解析 → 失败则回退LLM解析
│  input_processing │  截图 → 多模态LLM (Vision) OCR
└────────┬─────────┘
         │ 结构化对话 JSON
         ▼
┌──────────────────┐
│  上下文加载       │  用户画像 + 同目标历史会话 + 动态指令
│  user_store       │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  第一阶段LLM     │  结构化分析 → JSON
│  (低温度 0.3)     │  阶段/温度/意图/信号/用户问题
└────────┬─────────┘
         │ analysis JSON
         ▼
┌──────────────────┐
│  RAG检索         │  语义搜索 + 阶段过滤 + 标签匹配
│  vector_db       │  返回 top-5 知识块
└────────┬─────────┘
         │ retrieved knowledge
         ▼
┌──────────────────┐
│  第二阶段LLM     │  系统提示词 + 动态上下文 + 分析结果 + 知识 → 最终输出
│  (中温度 0.7)     │  诊断报告 + 3个回复选项 + 推荐 + 行动建议
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  输出 + 存储      │  返回给用户 + 保存会话记录
└──────────────────┘
```

## 关键设计决策

### 1. 为什么用两阶段LLM调用而不是一次调用

- 第一阶段 (分析) 用低温度确保判断一致性,输出结构化JSON供RAG检索使用
- 第二阶段 (生成) 用较高温度确保回复自然有创意
- 中间的RAG检索依赖分析结果的标签和信号,必须先完成分析

### 2. 为什么输入解析单独一个阶段

- 截图和文本粘贴的处理逻辑完全不同
- 解析可以用便宜的小模型 (gpt-4o-mini),分析和生成用大模型
- 解析失败时可以要求用户补充信息,不会浪费大模型token

### 3. 知识库为什么用四级分类

- 一级(场景)和二级(阶段)做硬过滤,缩小搜索范围
- 三级(类型)区分理论vs实操vs反面教材
- 四级(标签)做软匹配,与分析结果的detected_signals直接对应

### 4. 会话历史为什么要压缩

- 同一目标多次咨询时,早期细节不重要,趋势和关键转折才重要
- 压缩后的摘要占用更少token,留更多空间给当前对话和知识检索

## 成本估算 (单次分析)

| 阶段 | 模型 | 输入tokens | 输出tokens | 估算成本 |
|------|------|-----------|-----------|---------|
| 输入解析 | gpt-4o-mini | ~500 | ~300 | $0.0003 |
| 分析阶段 | gpt-4o | ~3000 | ~800 | $0.015 |
| 生成阶段 | gpt-4o | ~5000 | ~1500 | $0.030 |
| Embedding | text-embedding-3-small | ~200 | - | $0.00002 |
| **合计** | | | | **~$0.045/次** |

注: 如果使用Claude Sonnet替代gpt-4o,成本可降低约40%。

## 技术栈建议
